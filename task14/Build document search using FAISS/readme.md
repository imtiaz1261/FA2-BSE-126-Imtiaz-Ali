# Semantic Document Search using FAISS

A beginner-to-intermediate LLM/NLP project: load PDF/TXT documents, split
them into overlapping chunks, embed those chunks into dense vectors, store
them in a **FAISS** vector index (persisted to disk), and retrieve the
Top-K most semantically relevant chunks for any natural-language query.

## Features

- ‚úÖ Loads documents from a local folder ‚Äî supports **PDF** and **TXT**
- ‚úÖ Extracts text (PDF text extraction is page-aware)
- ‚úÖ Splits documents into chunks with **RecursiveCharacterTextSplitter**
  (chunk size 1000, overlap 200)
- ‚úÖ Generates embeddings via **Sentence-Transformers** (`all-MiniLM-L6-v2`),
  with optional **OpenAI Embeddings** support
- ‚úÖ Builds a **FAISS** vector index and **saves it locally** so it can be
  reloaded instantly without re-embedding
- ‚úÖ Interactive terminal search loop: Top-K retrieval (default K=5) with
  similarity score, document name, and **page number** for PDFs
- ‚úÖ Displays `"No relevant information found."` when nothing matches
- ‚úÖ Graceful error handling: missing/empty data folder, missing index,
  invalid query, unsupported file formats
- ‚úÖ Automatic offline fallback (TF-IDF + SVD) if you don't have internet
  access to download the Sentence-Transformers model weights
- ‚úÖ Clean, modular, PEP-8 compliant code

## Installation

```bash
# 1. Create and activate a virtual environment
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt
```

> **Note on the embedding model:** the first time you run the app,
> `sentence-transformers` downloads the `all-MiniLM-L6-v2` weights
> (~90 MB) from Hugging Face Hub, so you'll need internet access once.
> If you have no internet access at all, pass `--force-fallback` (see
> below) to use a fully local TF-IDF + SVD embedding instead ‚Äî every
> other part of the pipeline (chunking, FAISS indexing, page tracking,
> the search loop) behaves identically.

### Optional: using OpenAI embeddings instead

```bash
pip install langchain-openai
echo "OPENAI_API_KEY=sk-..." > .env
python app.py --use-openai
```

## Usage

### Build the index (run once, or whenever your documents change)

```bash
python ingest.py
```

Custom options:
```bash
python ingest.py --data-dir data --index-dir faiss_index \
                  --chunk-size 1000 --chunk-overlap 200
```

### Run the interactive search app

```bash
python app.py
```

The app automatically loads the existing `faiss_index/` if present, or
builds it on first run. Then type natural-language queries at the
`Search>` prompt.

In-app commands:

| Command       | Effect                                   |
|---------------|-------------------------------------------|
| `:k <N>`      | change the number of results (Top-K)     |
| `:exit`/`:quit` | leave the application                  |

Useful flags:
```bash
python app.py --rebuild             # force re-ingest even if an index exists
python app.py --top-k 3             # change default Top-K
python app.py --force-fallback      # no internet? use offline embeddings
python app.py --query "your question here"   # single non-interactive query
```

### Run a one-off query directly against the saved index

```bash
python search.py --query "what is chunking and why is overlap used" --top-k 5
```

## How It Works ‚Äî Component by Component

### 1. Document Loading (`ingest.py` ‚Üí `load_documents_from_folder`)
Scans `data/` for `.pdf` and `.txt` files. PDFs are loaded with LangChain's
`PyPDFLoader`, which produces **one Document object per page**, so page
numbers are preserved in metadata. TXT files are loaded with `TextLoader`
as a single Document (no page concept). Unsupported file types are
skipped with a warning rather than crashing the pipeline.

### 2. Text Extraction
Handled internally by `PyPDFLoader` (PDF) and `TextLoader` (TXT) as part
of the loading step above ‚Äî each returns a `page_content` string per
Document.

### 3. Chunking (`ingest.py` ‚Üí `split_documents`)
Uses LangChain's `RecursiveCharacterTextSplitter` with `chunk_size=1000`
and `chunk_overlap=200`. It tries to split on paragraph breaks first,
then sentences, then spaces ‚Äî only cutting mid-word as a last resort ‚Äî
so chunks stay semantically coherent. Each chunk inherits its parent
document's metadata (source filename, page number) plus a `chunk_id`.

### 4. Embedding Generation (`embeddings.py` ‚Üí `LocalEmbeddings`)
Implements LangChain's `Embeddings` interface (`embed_documents` /
`embed_query`). Primary backend is Sentence-Transformers'
`all-MiniLM-L6-v2` (384-dim, L2-normalized vectors). If that model can't
be loaded (no internet), it automatically fits a TF-IDF + Truncated SVD
embedding on your corpus instead, and **persists that fitted state to
disk** (`faiss_index/fallback_embedding_state.joblib`) so future queries
use the exact same vector space.

### 5. FAISS Index Creation (`ingest.py` ‚Üí `build_and_save_faiss_index`)
Calls `FAISS.from_documents(chunks, embedding_model)`, which embeds every
chunk and builds the index in memory, then `vectorstore.save_local(index_dir)`
writes `index.faiss` (the vector index) and `index.pkl` (the docstore /
metadata mapping) to disk.

### 6. Loading the Saved Index (`search.py` ‚Üí `load_faiss_index`)
`FAISS.load_local(index_dir, embedding_model, allow_dangerous_deserialization=True)`
reloads the index without recomputing any embeddings. Raises a clear
`FileNotFoundError` telling you to run `ingest.py` first if no index
exists yet.

### 7. Similarity Search (`search.py` ‚Üí `search_query`)
The user's query is embedded with the same embedding model, then
`vectorstore.similarity_search_with_score(query, k=top_k)` returns the
nearest chunks by FAISS L2 (Euclidean) distance. Since embeddings are
L2-normalized, that distance is converted to an intuitive 0‚Äì1
cosine-similarity-style score via `similarity = 1 - distance¬≤/2`.

### 8. Top-K Retrieval & Display
Results are formatted as `{rank, source, page, chunk_id, score, content}`
and printed with `utils.print_search_results()`. If the result list is
empty (or all results fall below an optional `--threshold`), the app
prints exactly: **`No relevant information found.`**

### 9. Search Workflow (`app.py`)
Ties it all together: build/load index ‚Üí interactive loop ‚Üí embed query
‚Üí FAISS search ‚Üí display results ‚Üí repeat until `:exit`.

## Error Handling Covered

| Scenario                          | Behavior                                                             |
|-------------------------------------|-------------------------------------------------------------------------|
| Missing `data/` folder              | `FileNotFoundError` with a clear message, exits cleanly                |
| Empty `data/` folder                | `ValueError`: "folder is empty, add .pdf/.txt files"                  |
| Missing FAISS index at search time  | `FileNotFoundError`: "Run ingest.py first"                            |
| Empty / whitespace-only query        | `ValueError`: "Query cannot be empty"                                  |
| Unsupported file format (e.g. .docx) | Skipped with a `WARNING` log, pipeline continues                      |
| No relevant results found            | Prints `"No relevant information found."`                             |
| Corrupted / unreadable file          | Logged as an error, that file is skipped, others still process        |
